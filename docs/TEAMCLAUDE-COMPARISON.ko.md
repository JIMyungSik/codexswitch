# TeamClaude 비교 감사

참고 프로젝트: [KarpelesLab/teamclaude](https://github.com/KarpelesLab/teamclaude)

## 이번에 적용한 설계

### 429 원인 분리

TeamClaude는 할당량 소진과 짧은 분당 제한을 구분한다. codexswitch도 다음 정책을 사용한다.

- 할당량 상태 헤더가 `rejected`이거나 기록된 사용량이 임계치를 넘으면 다음 계정으로 전환
- `retry-after`가 60초 이하면 같은 계정에서 한 번 기다렸다 재시도
- 원인이 불명확한 429는 다른 계정으로 연쇄 전파하지 않고 현재 계정을 잠시 정지

이 정책은 프롬프트 캐시를 유지하고 한 번의 요청 폭주가 모든 계정을 차례로 제한시키는 현상을 줄인다.

### 프록시 장애 시 점진적 복구

`cxs run --proxy`는 로컬 프록시를 먼저 확인한다. 실행 중이 아니면 경고 후 선택된 계정으로 직접 실행한다. 자동화에서 프록시가 필수라면 `--require-proxy`로 기존의 엄격한 실패 동작을 선택할 수 있다.

### 상태 가시성

`cxs server status`로 프록시 포트, 연결 가능 여부, 로테이션에 참여하는 계정 수를 확인한다.

### 출력 계층

TeamClaude TUI는 전체 화면을 계정, 진행 중 요청, 최근 완료 활동으로 나누고 로그가 화면을 끝없이 밀어내지 않게 한다. codexswitch는 Codex 답변 자체를 표시해야 하므로 다음과 같이 변형해 적용한다.

- `output auto`: 실제 터미널에서는 중간 명령·diff를 접고, 파이프와 CI에서는 원본 스트림 유지
- `output compact`: 환경과 관계없이 최종 답변과 작업 요약만 표시
- `output raw`: Codex 원본 스트림을 그대로 표시해 디버깅 가능

compact 화면은 진행 중에는 계정·세션·경과 시간 한 줄만 갱신하고, 완료 후 `Result`와 `Work summary`를 분리한다. 최종 답변은 Codex CLI의 `--output-last-message` 결과를 사용하므로 임의 문자열 파싱에 의존하지 않는다.

## 이미 codexswitch에 있는 동등·확장 기능

- 임계치 기반 로테이션과 우선순위
- 계정 enable/disable 및 사용량 유지
- 요청 단위 프록시 로테이션
- 실시간 사용량 헤더 수집
- 계정 전환 후 동일 Codex 세션 이어받기
- 공용·계정별 메모리
- 요청, 계정 흐름, 변경 파일을 연결한 작업 이력

## 보류한 기능

- MITM 인증서 프록시: Codex는 `chatgpt_base_url`을 지원하므로 현재는 복잡성과 인증서 위험이 이점보다 크다.
- 자동 전역 업데이트: 사용자 동의 없이 전역 npm 패키지를 변경하지 않는다.
- 주거용 외부 프록시: 계정 보안과 서비스 정책 위험 때문에 포함하지 않는다.
- 모델별 계정 라우팅: Codex 응답에서 안정적으로 모델별 별도 할당량을 확인할 수 있을 때 도입한다.
- storm ramp: 현재 프록시가 단일 Node 프로세스이지만 동시 요청 큐가 없다. 병렬 에이전트 사용량이 늘어나면 별도 동시성 게이트로 구현한다.
